🤖 让 Codex 用上国产大模型

MIMO × Codex CLI 代理方案 —— Responses API 到 ChatCompletions 的完整转换

🔧 技术实战 🔗 API 代理

💡 通俗模式

想象一下：你买了一个进口电器（Codex），但它只认一种特殊插座（Responses API）。而你家里的插座是另一种标准（ChatCompletions API）。怎么办？做一个转换插头（代理服务器），让进口电器能通过转换插头正常工作。

这个项目就是做了这样一个"转换插头"，让 OpenAI 的 Codex 编程工具能用上国产的 MIMO 大模型。

📋 背景

OpenAI 在 2025 年发布了 Codex CLI —— 一个在终端里运行的 AI 编程助手。它功能强大，能直接操作文件系统、执行命令、写代码。

⚠️ 问题来了

Codex CLI v0.133+ 只支持 Responses API（/v1/responses），而绝大多数第三方模型（包括小米 MIMO、DeepSeek、GLM 等）只支持 ChatCompletions API（/v1/chat/completions）。

💡 解决方案

在 Codex 和 MIMO 之间架设一个 FastAPI 代理服务器，实时翻译两种 API 的请求和响应。

Codex CLI ──── HTTP ────▸ Proxy (:8420) ──── HTTPS ────▸ MIMO API
请求翻译: Responses API → ChatCompletions API
响应翻译: ChatCompletions → Responses API events

🎯 核心设计

  • 请求翻译：将 Codex 发来的 items 数组（message / function_call / function_call_output）转换为 messages 数组
  • 响应翻译：将 MIMO 返回的 chat completions 响应转换为 SSE streaming events
  • 工具桥接：将 Codex 的 11 种工具（exec_command, write_stdin 等）映射为 function calling 格式
  • 推理映射：将 MIMO 的 reasoning_content 映射为 Responses API 的 reasoning output items

🔧 技术挑战

挑战 1：SSE 事件格式

Responses API 的 SSE 流中，每个事件的 data JSON 必须包含 type 字段。漏掉这个字段会导致 Codex 解析失败。

✅ 解决方案

在 make_event 函数中自动注入 type 字段：

def make_event(event_type: str, data: dict) -> str:
    # 确保每个事件数据都包含 type 字段
    if "type" not in data:
        data["type"] = event_type
    return f"event: {event_type}\ndata: {json.dumps(data)}\n\n"

挑战 2：工具调用转换

Codex 发送 function_call 类型的 input items，MIMO 返回 tool_calls 格式的响应。两者格式完全不同。

Step 1: 解析 Codex 的 input

将 function_call items 转换为 assistant + tool_calls messages

Step 2: 转发给 MIMO

MIMO 看到标准的 tool_calls 格式，正常处理

Step 3: 转换 MIMO 的响应

将 MIMO 返回的 tool_calls 转换为 Responses API 的 response.function_call_arguments.delta 和 .done events

# 将 Responses API input 转换为 ChatCompletions messages
def convert_input_to_messages(data: dict) -> tuple:
    messages = []
    instructions = data.get("instructions", "")
    if instructions:
        messages.append({"role": "system", "content": instructions})

    for item in data.get("input", []):
        if item.get("type") == "function_call":
            # Codex 的工具调用 -> assistant + tool_calls
            messages.append({
                "role": "assistant",
                "content": None,
                "tool_calls": [{
                    "id": item.get("call_id", f"call_{uuid4().hex[:8]}"),
                    "type": "function",
                    "function": {
                        "name": item.get("name", ""),
                        "arguments": item.get("arguments", "{}")
                    }
                }]
            })
        elif item.get("type") == "function_call_output":
            # 工具执行结果 -> tool role message
            messages.append({
                "role": "tool",
                "tool_call_id": item.get("call_id", ""),
                "content": item.get("output", "")
            })
    return messages, has_pending_tool_calls

挑战 3：推理内容映射

MIMO 的推理内容在 reasoning_content 字段中，而 Codex 期望通过 Responses API 的 reasoning output items 获取。

🧠 推理映射流程

MIMO streaming delta → 检测 reasoning_content 字段 → 生成 response.reasoning_summary_text.delta events → 最终生成 response.reasoning_summary_text.done event

挑战 4：多轮对话

Codex 在工具执行后会发送新一轮请求，包含之前的所有对话历史。代理需要正确保持上下文。

💡 关键洞察

Codex 的多轮对话通过 input 数组中的 items 来维护：message items 保存对话，function_call items 保存工具调用，function_call_output items 保存工具结果。代理只需要将这些 items 正确转换为 messages 数组即可。

📡 SSE 事件序列

一次完整的工具调用流程中，代理会生成以下 SSE 事件序列：

# 1. 开始响应
event: response.created
data: {"type":"response.created","response":{"id":"resp_xxx","status":"in_progress"}}

# 2. 添加输出项（工具调用）
event: response.output_item.added
data: {"type":"response.output_item.added","output_index":0,
       "item":{"type":"function_call","name":"exec_command","call_id":"call_xxx"}}

# 3. 参数流式传输
event: response.function_call_arguments.delta
data: {"type":"response.function_call_arguments.delta",
       "output_index":0,"argument_index":0,"delta":"{\"command\":\"ls\"}"}

# 4. 参数完成
event: response.function_call_arguments.done
data: {"type":"response.function_call_arguments.done",
       "output_index":0,"argument":"{\"command\":\"ls\"}"}

# 5. 输出项完成
event: response.output_item.done
data: {"type":"response.output_item.done","output_index":0,
       "item":{"type":"function_call","name":"exec_command"}}

# 6. 响应完成
event: response.completed
data: {"type":"response.completed","response":{"id":"resp_xxx","status":"completed"}}

🚀 安装配置

环境要求

  • macOS（使用 launchd 自启）
  • Python 3.8+
  • OpenAI Codex CLI（v0.133+）
  • 小米 MIMO API 密钥

步骤

1

安装依赖

pip install fastapi uvicorn httpx
2

配置代理

代理读取环境变量 XIAOMI_API_KEY 和 MIMO_BASE_URL，默认连接 https://token-plan-cn.xiaomimimo.com/v1。

3

配置 Codex

# ~/.codex/config.toml
[model]
provider = "custom"
base_url = "http://127.0.0.1:8420"
api_key = "any-key"
model = "mimo-v2.5-pro"
4

设置开机自启

使用 macOS 的 launchd 服务自动启动代理：

# ~/Library/LaunchAgents/com.xxx.codex-mimo-proxy.plist
# RunAtLoad: true, KeepAlive: true
# 自动在后台运行，无需手动启动

🎯 总结

通过一个轻量级的 FastAPI 代理，成功让 OpenAI Codex CLI 使用国产 MIMO 大模型。核心工作是处理两种 API 格式之间的差异：

  • SSE 事件格式（type 字段注入）
  • 工具调用格式（function_call ↔ tool_calls）
  • 推理内容格式（reasoning_content ↔ reasoning output items）
  • 多轮对话格式（items ↔ messages）

这个方案不仅适用于 MIMO，理论上可以适配任何只支持 ChatCompletions API 的模型，只需要修改目标 API 的地址和密钥即可。
